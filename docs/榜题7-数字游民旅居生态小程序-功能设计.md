# 榜题7：“数字游民”友好型旅居生态产品包与社群运营服务体系设计｜小程序功能设计

适用对象：挑战杯答辩与产品原型落地（MVP 可上线演示）

---

## 1. 产品定位与边界

### 1.1 一句话定位
面向“数字游民/旅居人群”的河源旅居入口：用“产品包 + 活动 + 社群 + 技能/项目对接”完成从获客到长期价值转化的闭环。

### 1.2 目标用户与角色
- 游客/数字游民（C 端）：找住、找工位、获取生活指引、报名活动、结识同伴、技能交换、参与共建
- 本地合作方（B 端轻量参与）：住宿/联合办公/活动组织者/社区点位/企业项目方（通过入驻与发布）
- 运营人员（平台方）：发布产品包与活动、审核内容、维护点位与合作方、拉新与社群运营、数据看板

### 1.3 MVP 范围（“能跑通”优先）
- 必做：产品包（展示+意向下单/预约）/活动报名/生活指引（地图+卡片）/技能交换&项目需求墙/社群入口与规则/基础数据统计
- 可选增强：支付闭环（真下单）/积分与任务体系/线下签到核销/合作方自助入驻/更细粒度的用户画像与推荐（AI 个性化）

## 3. 核心业务流程（答辩可讲的“闭环”）

### 3.1 答辩叙事主线（用一句话串起来）
我们用小程序把“来之前的决策”变成“来之后的连接”，再把“短暂停留”变成“可持续贡献”，最终形成“可量化、可运营、可变现”的旅居生态闭环。

### 3.2 五阶段闭环（每阶段 1 个抓手 + 1 个关键动作）
1) 获客：抓手=内容传播与落地页；关键动作=进入首页并点击核心入口  
2) 转化：抓手=产品包与活动双入口；关键动作=提交产品包意向/报名活动  
3) 留存：抓手=社群与画像；关键动作=加入社群并完善技能标签  
4) 贡献：抓手=技能交换与共建任务；关键动作=发布技能/需求并完成一次对接/任务  
5) 收益：抓手=线索撮合与票务/增值服务；关键动作=成交/核销并可复购

### 3.3 核心“双引擎”（为什么能既转化又留存）
- 转化引擎：产品包（住宿+联合办公+服务）解决“我来住哪、怎么工作、怎么省心”
- 留存引擎：活动+社群解决“我来和谁一起、在这能做什么、如何形成关系”

## 6. 指标闭环与功能设计（产品经理答辩版）

### 6.1 北极星指标（North Star）与分解

北极星指标（建议用于答辩与看板首页）：
- 月度“有效旅居人次”（EA: Effective Arrivals）= 当月到访且完成至少 1 次活动/社群互动的人数

为什么用它：
- 对齐政府与社区目标：把“人流”变成“参与”和“在地融合”
- 对齐商业可持续：到访与互动越高，线索成交、活动付费与复购越有基础

分解公式（用于解释闭环）：
- EA = UV × 转化率（意向/报名）× 到访率 × 互动率（社群/活动/发布）

### 6.2 五阶段指标闭环（指标口径 + 决策用途）

#### A. 获客（Acquisition：把人带进来）
核心目标：提高“有效访问”而不是只堆 UV  
关键指标：
- UV（按渠道拆分）：海报/短视频/公众号/转发
- 新客有效访问率 = 首次访问且停留≥30 秒或触发关键点击 / 新客 UV
- 首页关键入口 CTR = 点击产品包/活动/指引任一入口 / 首页 UV
产品抓手：
- 首页“产品包+活动”双推荐位、首次访问新手引导、可分享的活动海报卡片

#### B. 转化（Conversion：把意向变成到访）
核心目标：把“想来”变成“来过”  
关键指标：
- 产品包线索转化率 = 提交产品包意向 / 产品包详情 UV
- 活动报名转化率 = 报名 / 活动详情 UV
- 线索有效率 = 已联系（contacted）/ 已提交（submitted）
- 到访率（运营回填/核销）= 已到访（confirmed 或 attended）/ 已提交
产品抓手：
- 产品包详情“服务标准+权益”结构化展示、表单字段最小化、活动一键报名、客服/运营触点

#### C. 留存（Retention：让人留下来、愿意再来）
核心目标：让旅居从“单次消费”变成“持续关系”  
关键指标：
- D7 留存率（建议答辩重点）= 7 日内再次访问 / 首访用户
- 社群入群率 = 点击入群 / 到访用户
- 画像完成率 = 完善技能标签 / 到访用户
- 二次参与率 = 第二次活动报名或第二次意向提交 / 到访用户
产品抓手：
- “到访后引导”任务流（加入社群→选技能标签→报名下一场活动）、我的页聚合（我的活动/对接/任务）

#### D. 贡献（Contribution：把停留转成在地价值）
核心目标：让数字游民成为“资源与能力供给者”  
关键指标：
- 贡献渗透率 = 发布/对接/任务完成任一行为用户 / 活跃用户
- 对接成功率 = accepted / sent
- 共建任务完成率 = done / approved
- 本地项目参与人数（按项目维度汇总）
产品抓手：
- 技能/需求广场（模板化发布）、对接请求（站内沟通）、共建任务报名与状态流转

#### E. 收益（Revenue：让运营可持续）
核心目标：先跑通“可解释的收入逻辑”，再逐步接入支付与合同  
关键指标：
- 意向 GMV（先估算）= Σ（意向价格区间中位数 × 停留天数）
- 成交额（运营回填）与佣金收入（按合作协议比例）
- 活动付费收入（可选）与核销率
- 合作方数量（住宿/工位/活动）与活跃发布数
产品抓手：
- 线索状态管理与导出、活动票务（可选）、合作方入驻与上架（后期）

### 6.3 指标 → 功能映射（答辩时“一张表讲清楚为什么做这些功能”）

| 阶段 | 关键指标 | 对应功能模块 | 可运营动作 |
|---|---|---|---|
| 获客 | 新客有效访问率、入口 CTR | 首页推荐、新手引导、分享海报 | 渠道投放对比、落地页 A/B |
| 转化 | 线索/报名转化率、到访率 | 产品包详情+意向表单、活动报名、客服触点 | 表单减字段、活动“爆款”策划 |
| 留存 | D7 留存、入群率、画像完成率 | 社群入口、画像/技能标签、我的聚合 | 到访后任务流、社群分层运营 |
| 贡献 | 贡献渗透率、对接成功率 | 技能/需求发布、对接请求、共建任务 | 主题周（技能交换周）、任务激励 |
| 收益 | 成交额/佣金、付费收入 | 线索管理、核销/回填、票务（可选） | 合作方联营、套餐升级与加购 |

### 6.4 数据采集与看板结构（MVP 可落地）

埋点（必须有，支撑漏斗与渠道评估）：
- visit、view_home、click_home_entry、view_package_detail、submit_package_lead、view_event_detail、signup_event、join_group_click、complete_profile、create_post、match_sent、match_accepted、task_apply、task_done

回填（运营端必须有，支撑“到访/成交/核销”）：
- package_leads.status（submitted/contacted/confirmed/closed/lost）
- event_signups.status（signed/attended/cancelled）

看板（建议答辩展示 3 个层级）：
- 总览：UV、线索数、报名数、到访数、EA（北极星）、意向 GMV
- 漏斗：访问→详情→提交/报名→到访→互动（社群/发布/对接）
- 供给侧：产品包上架数、活动场次、合作方数量、点位覆盖

### 6.5 “两周一迭代”的增长实验清单（用于证明可持续运营）

- 获客：不同渠道海报与文案对比（看新客有效访问率与入口 CTR）
- 转化：产品包意向表单从 10 字段减到 6 字段（看线索转化率与有效率）
- 留存：到访后引导从“静态提示”改为“任务流”（看 D7 留存与画像完成率）
- 贡献：每周 1 场技能工作坊 + 广场置顶招募（看贡献渗透率与对接成功率）
- 收益：与 1–2 家住宿/工位做联营套餐（看到访率、成交额与复购）

## 8. AI 智配・数聚河源（选题 4）技术赋能设计

目标：把“产品包 + 活动 + 社群 + 技能/项目对接”的信息流做成“可推荐、可匹配、可量化”的转化引擎，让用户感觉更省心，让运营感觉更高效。

### 8.1 AI 在小程序端的可见功能（评审一眼能看懂）

1) 可解释推荐（首页/列表）
- 为你推荐产品包/活动/点位：每条卡片显示 1 行推荐理由（如“你选择了远程办公 + 安静偏好”“同技能的人最近报名了这场活动”）
- 支持“我不感兴趣/更喜欢××”的反馈按钮，用于个性化迭代

2) 智能匹配（技能/需求/项目对接）
- 在“项目详情/帖子详情”提供“推荐对接对象/相似需求”列表
- 先给出匹配理由（技能标签重合、时间可用、地理距离、过往互动），再允许一键发起对接

3) AI 旅居助手（生活指引问答）
- 入口放在生活指引页：可问“网速好的咖啡馆”“最近一周适合参加什么活动”“去万绿湖怎么走”
- 回答必须引用平台内信息（点位/活动/指引卡片），并提供一键导航/报名/收藏

4) 内容理解与风控（运营侧提效）
- 帖子/活动自动摘要与标签建议，减少人工编辑成本
- 低成本风险识别：联系方式外泄、涉敏内容、广告灌水，进入人工复审队列

5) 线索评分（运营侧提效）
- 对产品包意向线索给出 0-100 评分与原因（预算匹配度、到达日期临近、历史参与活跃度等）
- 运营端按评分排序，优先跟进高转化线索

### 8.2 AI 具体怎么实现（MVP 可落地的工程路径）

整体采用“规则 + 检索/向量 + 排序 + 可解释”的两段式方案，先保证效果可控、可上线演示，再逐步引入更复杂模型。

**A. 推荐与匹配：两段式（召回 → 排序）**
- 召回（Recall）：从 packages/events/pois/posts/tasks 中找候选
  - 规则召回：预算/时长/地点/时间窗口硬过滤
  - 内容召回：用 item_embeddings 做相似度检索（标签/描述向量化）
  - 行为召回：基于 event_logs 的“同类用户也喜欢”统计（MVP 可用简单共现）
- 排序（Rank）：对候选做打分，输出 TopN
  - 特征：用户画像（user_profiles）、实时意图（搜索词/筛选条件）、距离与时间、历史点击/报名/对接
  - 模型：MVP 可用加权打分或逻辑回归；进阶可用 GBDT/学习排序
- 可解释（Reason）：从贡献最大的 2–3 个特征生成 1 行理由并落到卡片上（写入 recommendation_impressions）

**B. 旅居助手：RAG（检索增强生成）**
- 知识源：guides/pois/events/packages 的结构化内容 + 运营配置的 FAQ
- 检索：先按 query 检索相关条目（关键词 + 向量检索）
- 生成：在“只允许引用检索到的条目”的约束下生成答案，并返回 citations（用于在小程序里点击跳转）

**C. 内容摘要/标签/审核：离线或半自动**
- 新发布内容进入 pending
- 自动给出 tags/summary/riskLevel（admin/ai/review）
- 运营一键采纳/修改后再 approved，上线可控

### 8.3 与现有功能的对齐关系（不改主结构，只增强“智能”）

- 首页推荐位：把“推荐”从人工置顶升级为“推荐 + 推荐理由”（对应 /api/recommendations/home）
- 技能/项目对接：在 posts/matches 之上补一个“候选推荐层”（对应 /api/match/suggest）
- 生活指引：在 guides/pois 之上补一个“问答入口”（对应 /api/ai/guide-chat）
- 运营漏斗：在 event_logs 的基础上补推荐曝光与反馈（recommendation_impressions），让“AI 是否有效”可量化

### 8.4 评估指标（答辩口径 + 可做实验）

- 推荐有效点击率：推荐卡片点击 / 推荐曝光
- 推荐转化率：推荐点击后提交意向/报名 / 推荐点击
- 匹配成功率：match_accepted / match_sent（已在 6.2D 提到，可直接纳入 AI 目标）
- 运营提效：平均跟进时长下降、线索有效率提升（contacted/submitted）

### 8.5 隐私与安全（评审常问点）

- 最小化采集：用户画像只用兴趣/技能/偏好等非敏字段；手机号等仅用于意向线索且可选
- 可控生成：旅居助手回答必须带 citations，避免“编造信息”
- 风控兜底：所有用户生成内容默认审核流转（pending→approved），可追溯（audit_logs）
